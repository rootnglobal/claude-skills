# 6. Git 버전 관리 규칙

> 테스트 통과 = 커밋. 모든 커밋은 '안전한 복원 지점'이다.

---

## 핵심 원칙

- **커밋 기준**: 전체 테스트가 통과한 순간에만 커밋한다
- **실패 상태 커밋 금지**: 테스트가 실패한 상태로는 절대 커밋하지 않는다
- **하나의 TODO = 하나의 커밋**: 여러 기능을 한 커밋에 넣지 않는다

---

## 커밋 메시지 규칙

| 접두사 | 용도 | 예시 |
|--------|------|------|
| `feat:` | 새 기능 구현 완료 | `feat: 사용자 로그인 기능 구현` |
| `fix:` | 버그 수정 완료 | `fix: BUG-003 토큰 만료 처리 누락` |
| `refactor:` | 리팩토링 (동작 변경 없음) | `refactor: auth 모듈 함수 분리` |
| `test:` | 테스트 추가/수정 | `test: 로그인 경계값 테스트 추가` |
| `docs:` | 문서 업데이트 | `docs: 2design.md 에러 처리 정책 추가` |
| `chore:` | 설정, 빌드 등 기타 | `chore: eslint 설정 추가` |

---

## 브랜치 전략

```
main ─────────────────────────── (안정, 태그)
  └── dev ────────────────────── (일상 개발, TDD 커밋)
        └── fix/BUG-NNN ──────── (버그 수정 전용, 머지 후 삭제)
```

- **main**: 항상 안정적인 상태. 마일스톤 태그를 이 브랜치에 붙인다.
- **dev**: 일상적인 개발 브랜치. TDD 사이클의 커밋이 여기에 쌓인다.
- **fix/BUG-NNN**: 버그 수정 전용 브랜치. 수정 완료 후 dev에 머지하고 삭제한다.

---

## 마일스톤 태그 전략

> 3todo.md의 Phase가 완료될 때마다 Git 태그를 붙인다. '큰 복원 지점'.

| 태그 | 시점 | 의미 |
|------|------|------|
| `v0.0.0` | 프로젝트 초기화 | git init, 초기 파일 구조 |
| `v0.1.0` | Phase 1 완료 | 스캐폴딩, 의존성, 로깅 모듈 세팅 |
| `v0.2.0` | Phase 2 완료 | 핵심 비즈니스 로직, 단위 테스트 통과 |
| `v0.3.0` | Phase 3 완료 | 모듈 간 통합 테스트 통과 |
| `v0.4.0` | Phase 4 완료 | 에지 케이스, 에러 처리 구현 |
| `v1.0.0` | Phase 5 완료 | 전체 테스트 통과, 리팩토링 완료, 배포 가능 |

## Git 운용 프롬프트

```
전체 테스트가 통과했어. 6gitops.md 커밋 규칙에 따라 커밋 메시지를 작성하고 커밋해. 
3todo.md에서 현재 Phase가 모두 완료되었으면 태그도 붙여.
```

```
AI가 코드를 잘못 수정해서 테스트가 깨졌어. 
마지막 통과 커밋으로 되돌리고 (git reset --hard), 다시 시도해.
```

---

## 롤백 절차

1. `git log --oneline -10`으로 최근 커밋 확인
2. 마지막 안전 커밋 해시 확인
3. `git reset --hard <커밋해시>`로 롤백
4. 전체 테스트 통과 확인
5. 다시 시도

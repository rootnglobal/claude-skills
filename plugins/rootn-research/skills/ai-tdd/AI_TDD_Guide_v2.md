## 1. 왜 AI 코딩에 설계가 필요한가

AI 코딩 어시스턴트는 강력하지만, 근본적인 한계가 있다. 맥락 창(context window)이 유한하고, 대화가 길어지면 초기 지시를 잊으며, 코드를 수정할 때 전체 시스템의 의존 관계를 완벽히 파악하지 못한다.

이 문제의 본질은 **AI의 기억이 휘발성**이라는 점이다. 사람은 머릿속에 전체 아키텍처를 들고 다니지만, AI는 매 턴마다 주어진 맥락만으로 판단한다. 설계 없이 AI에게 코딩을 시키면, 매번 '처음 보는 것처럼' 코드를 작성하고, 이전에 합의한 구조를 무시하거나 임의로 변경한다.

### 1.1 AI 코딩의 전형적 실패 패턴

- **맥락 소실**: 대화 30턴 이후 초기 요구사항을 잊어버림
- **구조 침식**: 버그 수정 시 다른 모듈의 인터페이스를 임의로 변경
- **해피 패스 편향**: 정상 흐름만 구현하고 에러/에지 케이스 무시
- **디버깅 불능**: 로그가 없거나 형식이 제각각이라 문제 원인 추적 불가
- **롤백 불가**: 버전 관리 없이 코드를 덮어쓰다 복구 불능 상태에 빠짐
- **테스트 부재**: '잘 되는 것 같다'며 검증 없이 다음 기능으로 이동

**해결책**: AI의 사고 과정을 파일로 외부화하고, Git으로 안전망을 깔고, 표준 로그로 디버깅 경로를 확보한다.

## 2. 핵심 원리: 파일 기반 맥락 외부화

> **핵심: 프롬프트 창에는 명령만, 맥락은 파일에 담는다.**

### 2.1 왜 파일인가

- **영속성**: 대화가 끊겨도 파일은 남는다. 새 세션에서 즉시 복원 가능.
- **검토 가능성**: 사람이 파일을 열어 AI의 계획을 검토하고, 오류를 사전에 잡을 수 있다.
- **버전 관리**: Git으로 설계 문서와 코드의 변경 이력을 함께 추적할 수 있다.
- **재사용성**: 비슷한 프로젝트에서 템플릿을 재사용할 수 있다.

### 2.2 프롬프트 3줄 원칙

프롬프트 창에는 가능하면 3줄 이상 쓰지 않는다. `.md` 파일을 참조하라는 명령만 한다.

```
1prd.md를 읽고, 2design.md 템플릿을 채워. 코드는 아직 작성하지 마.
```

```
3todo.md 다음 항목을 TDD로 구현해. 테스트 통과하면 git 커밋해.
```

```
7buglog.md에 버그 기록하고, 수정 후 전체 테스트 돌려. 통과하면 커밋.
```

## 3. 7-파일 프레임워크 구조

기존 5-파일 구조에서 표준 로깅 명세와 Git 운용 규칙을 추가한 7-파일 구조. 번호는 작업 순서이며, AI에게 읽는 순서의 신호를 준다.

| Phase | 파일명 | 역할 |
|-------|--------|------|
| 1 | `1prd.md` | 프로젝트 목표, 기능 목록, 제약 조건, 성공 기준 |
| 2 | `2design.md` | 아키텍처, 데이터 흐름, 모듈 인터페이스, 상태 관리 |
| 3 | `3todo.md` | 설계 기반 체크리스트, 단일 함수 단위 작업 |
| 4 | `4logging.md` | 로그 레벨, 포맷, 모듈별 로깅 규칙 명세 |
| 5 | `5test.md` | 기능별 테스트 케이스, 에지 케이스, 회귀 목록 |
| 6 | `6gitops.md` | 브랜치 전략, 커밋 규칙, 마일스톤 태그 |
| 7 | `7buglog.md` | 버그 기록, 근본 원인, 해결, 재발 방지 |

> **TIP**: 번호 = 순서 = 우선순위. 로깅(4)이 테스트(5)보다 앞선 이유는, 테스트 코드에서도 로깅 표준을 따라야 하기 때문이다.

## 4. Phase 1: 요구사항 정의

PRD(Product Requirements Document)는 '무엇을 만들 것인가'를 정의한다. AI가 방향을 잃을 때마다 이 파일을 참조하면 원래 의도를 복원한다.

### 4.1 PRD 필수 항목

- **프로젝트 개요**: 한 문단으로 이 프로젝트가 무엇인지 설명
- **핵심 기능 목록**: 구현할 기능을 P0/P1/P2 우선순위와 함께 나열
- **기술 스택**: 언어, 프레임워크, 라이브러리 명시
- **제약 조건**: 성능, 보안, 호환성 등 반드시 지켜야 할 규칙
- **성공 기준**: '완성'이라 판단하는 조건
- **비기능 요구사항**: 응답 시간, 에러율, 접근성 등

### 4.2 AI와 PRD 작성 대화법

PRD를 혼자 작성할 필요 없다. AI에게 '질문하게' 만든다.

```
나는 [프로젝트 설명]을 만들려고 해. 1prd.md 템플릿의 각 항목에 대해 나에게 질문해서 
요구사항을 구체화해줘. 한 번에 하나씩만 질문해.
```

> **TIP**: AI를 인터뷰어로 활용하면, 미처 생각하지 못한 요구사항까지 끌어낸다. 특히 에지 케이스, 에러 처리, 사용자 시나리오에서 효과적이다.

## 5. Phase 2: AI와의 설계 대화법

설계 단계는 이 방법론에서 **가장 중요**하다. PRD에서 '무엇을'이 정해졌다면, 설계에서는 '어떻게'를 결정한다.

> **핵심: 코드를 쓰기 전에 AI와 설계 대화를 충분히 나누는 것.**

### 5.1 설계 대화의 3단계

**Step 1: 아키텍처 탐색**

```
1prd.md를 읽고, 이 요구사항을 구현할 수 있는 아키텍처를 3가지 제안해줘. 
각각의 장단점과 트레이드오프를 설명해. 코드는 쓰지 마.
```

**Step 2: 구조 확정**

```
선택한 아키텍처를 기반으로 모듈 구조를 확정해줘. 각 모듈의 책임, 공개 인터페이스
(함수명, 인자, 반환값), 모듈 간 의존 관계를 명시해. 2design.md에 기록해. 코드는 아직 작성하지 마.
```

**Step 3: 위험 요소 점검**

```
2design.md를 다시 읽고, 이 설계에서 발생할 수 있는 문제점을 5가지 찾아줘. 
각 문제에 대한 대응 방안도 제시해. 2design.md의 '위험 요소' 섹션에 추가해.
```

### 5.2 설계 대화 시 주의점

- **코드 작성 금지를 명시적으로 선언**: '코드는 쓰지 마'를 반드시 포함
- **한 턴에 하나의 주제만**: 아키텍처와 데이터 모델을 동시에 논의하면 맥락 희석
- **결정 사항은 즉시 파일에 기록**: 합의 내용은 바로 2design.md에 반영
- **반박을 요청**: '이 설계의 약점을 공격해봐'로 비판적 검토를 유도

## 6. Phase 3: 상세 설계 명세서

설계 대화의 결과물이 `2design.md`이다. AI가 코딩할 때 반드시 따라야 할 **'설계 계약서'**이다.

> **핵심: 빈칸 채우기 형식의 템플릿으로 만들어, AI가 모든 항목을 강제로 검토하게 한다.**

### 6.1 설계 명세서 필수 섹션

- **시스템 아키텍처**: 전체 구조를 한 문단으로 설명. 사용 패턴(MVC, 이벤트 드리븐 등) 명시.
- **데이터 흐름**: 입력 → 처리 → 출력의 전체 경로를 단계별로 기술.
- **모듈 인터페이스 정의**: 각 모듈이 노출하는 함수의 시그니처(이름, 인자 타입, 반환 타입) 확정. 모듈 간 '계약'.
- **상태 관리 규칙**: 상태를 어디에 보관하고, 누가 변경 권한이 있는지 명시.
- **에러 처리 정책**: 예외 상황별 대응을 미리 정의. 재시도, 폴백, 사용자 알림 전략 명시.
- **로깅 통합 지점**: 각 모듈이 4logging.md의 어떤 로그 레벨을 사용하는지 명시. 설계 단계에서 로깅을 결정.
- **위험 요소와 대응**: 설계상 잠재적 문제점과 완화 전략 기록.

### 6.2 상세 명세서를 길게 쓰는 이유

- **AI는 암묵지를 모른다**: 사람에게 '당연한' 것도 AI에게는 명시적으로 알려줘야 한다.
- **모호함 = 버그**: 설계에 빈 곳이 있으면 AI가 임의로 해석한다.
- **새 세션 대비**: 대화가 끊기고 새 세션을 열 때, 긴 설계 문서만 읽으면 전체 맥락이 복원된다.

## 7. Phase 4: 표준 로깅 모듈 설계

AI가 만든 코드에서 버그가 발생했을 때, 로그가 없으면 원인을 찾을 수 없다. AI는 `console.log('here')`나 `print('debug')` 같은 임시 로그를 남기는 경향이 있고, 배포 전에 이를 제거해버린다.

### 7.1 왜 로깅을 별도 Phase로 분리하는가

- **디버깅 시간 단축**: 구조화된 로그가 있으면 AI가 버그 원인을 스스로 추적할 수 있다.
- **AI의 자가 진단**: '에러 로그를 보고 원인을 분석해줘'라는 프롬프트가 가능해진다.
- **일관성**: 모든 모듈이 같은 포맷으로 로그를 남기면, 로그 검색과 필터링이 쉬워진다.
- **테스트보다 먼저**: 테스트 코드에서도 로깅 표준을 따라야 하므로, 테스트 설계(Phase 5)보다 먼저 정의한다.

### 7.2 로그 레벨 정의

| 레벨 | 용도 | 예시 |
|------|------|------|
| DEBUG | 개발 중 상세 추적 정보 | 함수 진입/퇴장, 변수값 변화 |
| INFO | 정상 동작 확인 이벤트 | 서버 시작, 요청 처리 완료, 사용자 로그인 |
| WARN | 잠재적 문제, 당장 에러는 아님 | 응답 시간 임계치 초과, 재시도 발생 |
| ERROR | 기능 실패, 복구 가능 | API 호출 실패, 데이터 유효성 검증 실패 |
| FATAL | 시스템 중단 수준의 치명적 오류 | DB 연결 불가, 필수 설정 파일 누락 |

### 7.3 표준 로그 포맷

```
[YYYY-MM-DD HH:mm:ss.SSS] [LEVEL] [MODULE] [FUNCTION] MESSAGE {context}
```

실제 출력 예시:

```
[2026-04-13 14:23:01.456] [ERROR] [auth] [validateToken] 토큰 검증 실패 {userId: 'u123', reason: 'expired', expiredAt: '2026-04-13T12:00:00Z'}
```

핵심은 `[MODULE]`과 `[FUNCTION]`을 반드시 포함하는 것이다. 마지막의 `{context}`에는 디버깅에 필요한 관련 변수값을 JSON으로 첨부한다.

### 7.4 로깅 규칙

- **함수 진입/퇴장 로그**: 핵심 비즈니스 함수는 진입 시 DEBUG, 퇴장 시 DEBUG + 결과를 로깅
- **에러는 반드시 context 포함**: ERROR 이상은 관련 변수값, 입력값, 스택 정보를 context에 첨부
- **외부 호출 전후 로그**: API 호출, DB 쿼리 등 외부 시스템 호출은 호출 전(INFO)과 응답 후(INFO/ERROR) 로깅
- **민감 정보 마스킹**: 비밀번호, 토큰, 개인정보는 로그에 남기지 않거나 마스킹 처리
- **성능 측정 로그**: 주요 처리의 소요 시간을 INFO 레벨로 기록 (예: `DB 쿼리 완료 {elapsed: 142ms}`)

### 7.5 AI에게 로깅을 시키는 프롬프트

```
4logging.md를 읽고, 지금 구현하는 모든 함수에 로깅 표준을 적용해. 
console.log나 print는 절대 사용하지 마. 반드시 로거 모듈을 통해서만 로그를 남겨.
```

```
에러가 발생했어. 로그를 확인하고, [ERROR] 레벨 로그에서 context를 분석해서 원인을 찾아줘.
```

> **TIP: 로그 기반 디버깅 루프** — 버그가 발생하면 '로그를 보여줘 → 로그를 분석해줘 → 원인을 찾아 수정해줘'의 3단계 프롬프트로 AI가 스스로 디버깅하게 할 수 있다. 로그가 표준화되어 있어야 이 루프가 작동한다.

## 8. Phase 5: TDD 사이클 운용

TDD(Test-Driven Development)는 '테스트를 먼저 작성하고, 그 테스트를 통과하는 코드를 구현'하는 방법론이다. AI 코딩에서 TDD는 단순한 모범 사례가 아니라, **버그를 억제하는 가장 효과적인 안전망**이다.

### 8.1 AI TDD의 3-스텝 사이클

| 단계 | 설명 |
|------|------|
| **RED** | 테스트 먼저 작성. 이 시점에서 테스트는 반드시 실패해야 한다. |
| **GREEN** | 테스트를 통과하는 최소한의 코드만 작성. 통과만 하면 된다. |
| **REFACTOR** | 테스트 통과 상태를 유지하면서 코드 정리. 2design.md 규칙 준수. |

REFACTOR가 끝나면 **Git 커밋**. 이것이 Phase 6과의 연결점이다.

### 8.2 AI에게 TDD를 시키는 프롬프트

```
3todo.md에서 다음 미완료 항목을 확인해. 5test.md에서 해당 항목의 테스트 케이스를 먼저 작성해. 
테스트가 실패하는 것을 확인한 뒤, 최소한의 코드로 통과시켜. 4logging.md 표준에 맞게 로그도 넣어. 
통과하면 전체 테스트 돌리고, 회귀 없으면 git 커밋해.
```

### 8.3 테스트 케이스 설계 원칙 (4가지 범주)

- **정상 경로 (Happy Path)**: 기본적인 성공 시나리오 검증
- **경계값 (Boundary)**: 최소값, 최대값, 빈 입력, null 등
- **에러 케이스 (Error)**: 잘못된 입력, 네트워크 실패, 권한 부족 등
- **상태 전이 (State)**: 상태 변경 시나리오 (로그인→로그아웃, 생성→수정→삭제)

## 9. Phase 6: Git 마일스톤 버전 관리

Git은 이 워크플로우의 **안전망**이다. AI가 코드를 잘못 수정했을 때 즉시 되돌릴 수 있고, 테스트가 통과한 시점마다 '안전한 복원 지점'을 만들어둔다.

### 9.1 핵심 원칙: 테스트 통과 = 커밋

커밋의 기준은 단 하나: **전체 테스트가 통과한 순간**. 테스트가 실패한 상태로는 절대 커밋하지 않는다. 모든 커밋이 '안전한 복원 지점'이 된다.

### 9.2 커밋 메시지 규칙

| 접두사 | 용도 | 예시 |
|--------|------|------|
| `feat:` | 새 기능 구현 완료 | feat: 사용자 로그인 기능 구현 |
| `fix:` | 버그 수정 완료 | fix: BUG-003 토큰 만료 처리 누락 |
| `refactor:` | 리팩토링 (동작 변경 없음) | refactor: auth 모듈 함수 분리 |
| `test:` | 테스트 추가/수정 | test: 로그인 경계값 테스트 추가 |
| `docs:` | 문서 업데이트 | docs: 2design.md 에러 처리 정책 추가 |
| `chore:` | 설정, 빌드 등 기타 | chore: eslint 설정 추가 |

### 9.3 브랜치 전략

- **main**: 항상 안정적인 상태. 마일스톤 태그를 이 브랜치에 붙인다.
- **dev**: 일상적인 개발 브랜치. TDD 사이클의 커밋이 여기에 쌓인다.
- **fix/BUG-NNN**: 버그 수정 전용 브랜치. 수정 완료 후 dev에 머지하고 삭제한다.

### 9.4 마일스톤 태그 전략

| 태그 | 시점 | 의미 |
|------|------|------|
| `v0.1.0` | Phase 1 기반 구조 완료 | 프로젝트 스캐폴딩, 의존성, 로깅 모듈 세팅 완료 |
| `v0.2.0` | Phase 2 핵심 기능 완료 | 핵심 비즈니스 로직 구현, 단위 테스트 통과 |
| `v0.3.0` | Phase 3 통합 완료 | 모듈 간 통합 테스트 통과 |
| `v0.4.0` | Phase 4 에러 처리 완료 | 에지 케이스, 에러 처리 구현 |
| `v1.0.0` | Phase 5 마무리 완료 | 전체 테스트 통과, 리팩토링 완료, 배포 가능 |

### 9.5 AI에게 Git을 시키는 프롬프트

```
전체 테스트가 통과했어. 6gitops.md 커밋 규칙에 따라 커밋 메시지를 작성하고 커밋해. 
3todo.md에서 현재 Phase가 모두 완료되었으면 태그도 붙여.
```

```
AI가 코드를 잘못 수정해서 테스트가 깨졌어. 마지막 통과 커밋으로 되돌리고 (git reset --hard), 다시 시도해.
```

> **WARNING**: 커밋 없이 진행하면, AI가 A를 구현하다 B를 부수고, B를 고치다 C를 부수면, 되돌릴 지점이 없다.

## 10. Phase 7: 버그 추적과 회귀 방지

버그는 반드시 발생한다. 문제는 **같은 버그가 반복**되는 것이다. `7buglog.md`는 AI가 과거 실수에서 학습하도록 만드는 **'오류 기억 장치'**이다.

### 10.1 버그 기록 형식

```markdown
## BUG-001: [버그 제목]
- 발생일: YYYY-MM-DD
- 관련 로그: [ERROR] 로그에서 복사한 해당 로그 라인
- 증상: 어떤 현상이 발생했는가
- 근본 원인: 왜 발생했는가 (5 Why 분석)
- 수정 내용: 어떻게 고쳤는가
- 영향 범위: 이 수정이 다른 모듈에 미치는 영향
- 재발 방지: 이 유형의 버그를 막기 위한 규칙
- 추가된 테스트: 5test.md 회귀 테스트 섹션에 추가한 테스트
- Git: fix/BUG-001 브랜치에서 수정, 커밋 해시 기록
```

### 10.2 버그 수정 워크플로우

1. **브랜치 생성**: `fix/BUG-NNN` 브랜치를 만든다
2. **로그 확인**: `[ERROR]` 레벨 로그를 분석하여 원인을 파악
3. **재현 테스트**: 버그를 재현하는 테스트를 5test.md 회귀 섹션에 추가
4. **수정**: 최소 범위로 수정. 다른 파일은 건드리지 않는다
5. **전체 테스트**: 모든 테스트가 통과하는지 확인
6. **기록**: 7buglog.md에 버그를 기록
7. **커밋/머지**: fix 브랜치를 dev에 머지하고, 브랜치를 삭제

> **TIP**: 같은 패턴의 버그가 3회 이상 반복되면, 그 패턴을 `2design.md`의 규칙으로 승격시킨다. 예: '비동기 호출에서 에러 처리 누락'이 반복 → 2design.md에 '모든 비동기 호출은 try-catch 필수' 규칙 추가.

## 11. 통합 워크플로우: 전체 사이클

### 11.1 프로젝트 시작 (1회)

1. `1prd.md`에 요구사항을 정의한다 (AI와 대화하며 작성)
2. `2design.md`에 아키텍처와 상세 설계를 확정한다 (코드 금지)
3. `4logging.md`에 로깅 표준을 정의한다
4. `3todo.md`에 작업을 분해한다 (단일 함수 단위)
5. `5test.md`에 테스트 케이스를 설계한다
6. `6gitops.md`에 Git 규칙을 확정한다
7. `git init`, 초기 커밋, `v0.0.0` 태그

### 11.2 구현 루프 (반복)

```
3todo.md 미완료 항목 확인
  → 5test.md에서 테스트 먼저 작성 (RED)
  → 최소 코드로 구현 + 4logging.md 표준 로깅 적용 (GREEN)
  → 2design.md 규칙 준수하며 리팩토링 (REFACTOR)
  → 전체 테스트 통과 확인
  → 6gitops.md 규칙에 따라 커밋
  → 3todo.md 체크 표시
  → Phase 완료 시 태그 (v0.X.0)
```

### 11.3 버그 발생 시 (분기)

```
버그 발생
  → fix/BUG-NNN 브랜치 생성
  → 4logging.md 로그로 원인 분석
  → 5test.md에 재현 테스트 추가
  → 최소 범위 수정
  → 전체 테스트 통과
  → 7buglog.md에 기록
  → 커밋, dev에 머지
```

### 11.4 세션 복구 (대화 끊김 후)

```
1prd.md, 2design.md, 3todo.md를 순서대로 읽고 현재 상황을 요약해줘. 
4logging.md와 6gitops.md 규칙도 확인해. git log --oneline -10으로 최근 커밋을 보여줘. 
그 다음 3todo.md에서 미완료 항목을 이어서 작업해.
```

## 12. 프롬프트 패턴 라이브러리

상황별로 복사해서 쓸 수 있는 프롬프트 모음.

| 상황 | 프롬프트 |
|------|----------|
| **프로젝트 시작** | `1prd.md 템플릿을 읽고 각 항목에 대해 나에게 질문해서 요구사항을 완성해줘.` |
| **설계 시작** | `1prd.md를 읽고, 2design.md 템플릿의 모든 항목을 채워. 코드는 작성하지 마.` |
| **설계 검증** | `2design.md를 다시 읽고, 약점 5가지를 찾아서 대응 방안과 함께 위험 요소 섹션에 추가해.` |
| **로깅 세팅** | `4logging.md를 읽고, 로거 모듈을 먼저 구현해. 모든 코드는 이 로거만 사용해.` |
| **작업 분해** | `2design.md를 기반으로 3todo.md를 작성해. 각 항목은 단일 함수 단위로 쪼개.` |
| **TDD 실행** | `3todo.md 다음 미완료 항목에 대해 테스트 먼저 → 최소 구현(로깅 포함) → 전체 테스트 → 커밋.` |
| **버그 수정** | `fix/BUG-NNN 브랜치를 만들고, 로그를 분석해서 원인을 찾아. 재현 테스트 추가 → 수정 → 전체 테스트 → 7buglog.md 기록 → 커밋.` |
| **세션 복구** | `1prd.md, 2design.md, 3todo.md를 읽고 현재 상황 요약해. git log -10도 보여줘. 이어서 작업해.` |
| **범위 제한** | `이 파일만 수정해: [파일명]. 다른 파일은 절대 건드리지 마.` |
| **마일스톤** | `3todo.md에서 현재 Phase가 모두 완료됐어. 태그 v0.X.0을 붙여줘.` |
| **롤백** | `마지막 커밋이 잘못됐어. git reset --hard HEAD~1 하고 다시 시도해.` |
| **로그 디버깅** | `에러가 발생했어. [ERROR] 로그를 확인하고 context를 분석해서 원인을 찾아줘.` |

## 13. 안티패턴과 트러블슈팅

### 13.1 피해야 할 안티패턴

- **한 번에 전체 구현 요청**: '전체 앱을 만들어줘'라고 하면 구조 없이 거대한 코드 덩어리를 생성한다. 모듈 단위로 쪼개서 요청할 것.
- **설계 없이 TODO로 직행**: PRD에서 바로 TODO로 가면 AI가 매번 다른 구조로 코딩한다. 2design.md가 앵커 역할.
- **console.log 방치**: AI가 디버깅용 console.log를 남기면 진짜 로그와 섞여서 분석 불가능. 4logging.md 표준 로거만 사용.
- **커밋 없이 다음 기능으로**: 테스트 통과한 순간이 곧 커밋 시점. 커밋 없이 넘어가면 롤백 불가능.
- **큰 커밋**: 여러 기능을 한 커밋에 넣으면 문제 발생 시 어떤 변경이 원인인지 찾기 어렵다. 하나의 TODO 항목 = 하나의 커밋.
- **버그 수정 후 기록 생략**: 같은 버그가 3번째 반복될 때, 7buglog.md가 있었다면 첫 번째에서 잡았을 것.

### 13.2 자주 발생하는 문제와 해결법

| 문제 | 해결법 |
|------|--------|
| AI가 설계 문서를 무시하고 독자적으로 구현 | '2design.md의 인터페이스 정의를 반드시 따라'를 프롬프트에 명시. 구현 후 인터페이스 일치 여부를 별도로 검증. |
| AI가 로그를 안 넣거나 형식을 안 따름 | '4logging.md를 다시 읽고, 방금 구현한 함수에 로깅을 추가해'라고 별도 프롬프트를 보낸다. |
| Git 커밋을 까먹음 | TDD 프롬프트에 '테스트 통과하면 커밋해'를 항상 포함. 습관이 될 때까지 명시. |
| 새 세션에서 맥락 복원이 불완전 | 1prd, 2design, 3todo를 읽고 요약하라고 먼저 시킨다. git log -10도 함께 보게 한다. |

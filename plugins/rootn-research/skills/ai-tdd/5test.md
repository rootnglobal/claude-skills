## 테스트 설계 원칙

모든 기능은 아래 4가지 범주의 테스트 케이스를 갖는다:

| 범주 | 설명 | 예시 |
|------|------|------|
| **정상 경로 (Happy Path)** | 기본적인 성공 시나리오 검증 | 올바른 입력 → 기대 출력 |
| **경계값 (Boundary)** | 최소값, 최대값, 빈 입력, null 등 | 빈 배열, 0, MAX_INT |
| **에러 케이스 (Error)** | 잘못된 입력, 네트워크 실패, 권한 부족 등 | 잘못된 타입, timeout |
| **상태 전이 (State)** | 상태 변경 시나리오 | 생성 → 수정 → 삭제 |

## TDD 실행 규칙

1. **RED**: 테스트를 먼저 작성. 실행하면 반드시 실패해야 한다.
2. **GREEN**: 테스트를 통과하는 최소한의 코드만 작성.
3. **REFACTOR**: 테스트 통과 상태를 유지하면서 코드 정리. `2design.md` 규칙 준수.
4. **COMMIT**: REFACTOR 완료 후 전체 테스트 통과 확인 → Git 커밋.

## TDD 프롬프트

```
3todo.md에서 다음 미완료 항목을 확인해. 5test.md에서 해당 항목의 테스트 케이스를 먼저 작성해. 
테스트가 실패하는 것을 확인한 뒤, 최소한의 코드로 통과시켜. 4logging.md 표준에 맞게 로그도 넣어. 
통과하면 전체 테스트 돌리고, 회귀 없으면 git 커밋해.
```

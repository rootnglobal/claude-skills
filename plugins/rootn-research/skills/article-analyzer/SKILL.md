# Article Analyzer — 기사 투자자 관점 분석

> **공통 규칙**: 작업 시작 전 `_shared/common-rules.md`(로컬 실행 시 스킬 폴더 기준 `../_shared/common-rules.md`)를 읽고 전 구간에 적용한다 — fail-fast, 자동진행/사전확인, ASCII 파일명, 드라이브 3점 검증, 신뢰도 표기, 수집기간 기본 2년. 그리고 **태스크 완료 후 스킬 피드백 루프(§12)**: 최종 산출물 전달 직후(태스크 단위 1회) 새로 확인된 소스·URL·우회법, 반복된 실패 패턴, 사용자가 수동으로 고친 부분을 점검한다. 개선점이 있으면 업로드본 SKILL.md를 즉석 수정하고 동일 diff를 SSOT repo(claude-skills)에 반영한다 (repo 미접근 환경이면 diff를 출력해 "다음 Cowork 세션에서 repo 반영 필요"로 flagging). 없으면 "스킬 업데이트 없음" 한 줄로 종료한다.


기사 1건(또는 소수)을 통독하고 "이게 내 커버리지·포트폴리오에 무슨 의미인가"까지 해석하는 스킬. 단순 요약이 아니라 **실적·밸류에이션에 닿는 경로**를 추론하는 것이 목적이다.

> **모델 전제**: 이 스킬은 Opus 계열 모델에서 실행을 전제로 한다. 기사 전문을 통독하고(발췌 스킵 금지), 표면 요약에 그치지 말고 P×Q·마진·멀티플 경로 추론, 2차 파급(경쟁사·supply chain), 사실/추측 구분까지 수행하라. 절차를 기계적으로 따르기보다 아래 원칙을 판단 기준으로 삼아 기사 성격에 맞게 적용한다.

## Step 0. 입력 판별

| 입력 형태 | 처리 |
|---|---|
| PDF·이미지·스크린샷 첨부 | 직접 읽기(이미지는 vision으로 통독). 컨텍스트에 이미 내용이 있으면 재로드 금지 |
| 붙여넣은 텍스트 | 그대로 분석 |
| URL | Step 1의 확보 프로토콜 |
| 기사 여러 건 | 각 기사를 개별 분석 후, 마지막에 "공통 시사점/상충 지점" 통합 섹션 추가 |

기사 언어는 무엇이든(한/영/일/중) 그대로 읽는다. 번역본을 찾으러 가지 않는다.

## Step 1. 기사 확보 — URL 접근 실패 시 대체 소스 프로토콜

1. `web_fetch`로 원문 시도. **최대 2회**(재시도 1회). 페이월·로그인·CAPTCHA·robots 차단·본문 잘림이면 즉시 포기하고 2로.
2. 부분 확보라도 됐으면(제목·리드·앞 몇 문단) 그것을 검색 단서로 쓴다. 아예 실패면 URL slug·사용자가 말한 내용에서 키워드를 뽑는다.
3. **같은 '사건'을 다룬 대체 매체 검색** (같은 기사의 재게재만 찾는 게 아님):
   - 검색어 = 핵심 고유명사 + 사건 키워드. **기사 언어(현지어)로 먼저 검색**, 안 나오면 영어.
   - 통신사(Reuters·Bloomberg·연합·共同·新華) 발 기사는 재게재 매체가 많으니 헤드라인 문구로 검색하면 잘 잡힌다.
   - 접근 가능한 대체 소스 2~3개를 읽는다. 각 소스도 fetch 최대 2회, 막히면 다음 후보.
   - **범위 제한 — 폴백의 목적은 원문 근사 재현이지 토픽 브리핑이 아니다**: 검색은 slug/제목의 **논지 키워드를 유지**한 1회 + 보완 1회까지만. 대체 소스는 **원문과 같은 각도 + 기사일 ±3일 이내 보도**로 한정하고, 그보다 오래된 배경 기사·과거 경위 기사는 인용하지 않는다. 배경까지 원하면 사용자가 요청할 때만 확장한다. (소스 간 상충 숫자 병기는 유지.)
   - URL에 내용 단서가 전혀 없고(해시형 slug) URL 문자열 검색도 매칭이 없으면 **추측으로 기사를 골라 분석하지 말 것**. 실패 경로를 보고하고 제목/주제 한 줄을 요청한다.
4. 대체 소스로 분석했으면 출력 최상단에 명시: **"⚠ 원문(매체명, URL) 접근 불가(사유). 아래는 [대체 매체 1·2]의 동일 사건 보도 기준."** 대체 보도가 원문과 다른 각도/추가 사실을 담았을 수 있음을 감안해, 소스 간 서로 다른 숫자·주장이 있으면 병기한다.
5. 대체 소스도 전부 실패하면 멈추지 말고, 확보된 단편(제목·리드)만으로 "확인된 범위 내 분석" + 무엇을 확인 못 했는지 명시.

## Step 2. 종목·이해관계 매핑

- 기사에 등장하는 **상장사·티커**를 식별(직접 언급 + 명백한 관련사). 비상장사면 상장 모회사·경쟁사·고객사로 연결.
- 종목 기사가 아니어도(매크로·정책·산업) **어느 섹터/국가의 어떤 종목군에 닿는가**를 반드시 한 단계 구체화한다. "반도체에 영향" 수준 금지 — 장비/소재/파운드리 중 어디, P인지 Q인지까지.
- 사용자의 커버리지(일본·한국·미국·중국 주식) 관점에서 우선순위를 매긴다.

## Step 3. 투자자 관점 분석

기사를 아래 렌즈로 해체한다. 해당 없는 항목은 건너뛴다(빈 섹션 채우기 금지).

1. **무슨 일인가** — 사건을 한 문장으로. 날짜·주체·행위·규모.
2. **핵심 숫자** — 금액·물량·단가·%·기간·가이던스. 기사 속 숫자는 전부 건져낸다. 단위·통화·기준시점 명시.
3. **실적 경로** — 이 사건이 어느 회사의 P(단가)·Q(물량)·마진·CAPEX·멀티플 중 무엇을, 언제부터(분기 단위), 얼마나 움직이는가. 가능하면 back-of-napkin: 매출 대비 몇 %인지, 컨센서스 대비 증분인지.
4. **2차 파급** — 경쟁사(반사이익/피해), 고객·공급사(supply chain), 대체재. 승자/패자 구도.
5. **사실 vs 해석 분리** — ① 확인된 사실(공시·발표 인용) ② 기사 작성자/애널리스트의 해석 ③ 익명 소식통·관측. 각각 구분해 표기. 회사 부인 코멘트가 있으면 반드시 포함.
6. **신뢰도 태깅** — 소스별 高/中/低 + 단독/복수 보도 여부. 단독 보도·익명 소식통 기반이면 "신뢰도 낮음, 후속 확인 필요"로 명시하되 버리지 않는다.
7. **타이밍** — 기사 작성일 기준으로 해석(오래된 기사면 그 이후 전개 여부를 웹서치 1회로 확인해 "이후 업데이트" 한 줄 추가). 주가가 이미 반영했을 법한지 여부도 짚는다.
8. **후속 확인 포인트** — 다음 공시·실적발표·이벤트에서 무엇을 확인해야 하는가. 커버 종목이면 IR 질문 후보 1~3개(부드러운 표현).

## Step 4. 출력

**기본 산출물은 채팅 응답**(파일 생성 안 함). 국문. 아래 뼈대를 기사 성격에 맞게 가감:

```
[⚠ 대체 소스 고지 — 해당 시에만]
■ 한 줄 요약
■ 핵심 사실·숫자 (표 또는 불릿, 출처·날짜 포함)
■ 투자 시사점 — 실적 경로 (종목별)
■ 2차 파급 (경쟁사·supply chain)
■ 사실/해석/관측 구분 + 신뢰도
■ 후속 확인 포인트 (커버 종목이면 IR 질문 후보)
출처: [매체명, 기사일자, URL]
```

- 원문 언어의 핵심 표현·인용·고유명사·숫자는 원어 병기(재번역 손실 방지 — 특히 IR 질문 후보에 쓸 문구).
- 저작권: 내부 리서치 용도. 기사 문장을 통째로 옮기지 말고 **요약 + 출처 링크**. 직접 인용은 짧은 구절만.

### 저장 (요청 시에만)

사용자가 "저장해줘/드라이브에 올려줘" 하면: 종목이 식별된 경우 `기사분석_[종목]_[YYMMDD].md`(ASCII 파일명 — 종목은 영문명/티커)로 작성해 드라이브 `Investment Research/[영문명 (코드)]/리서치/`에 업로드. news-collector digest와 동일하게 "investment-note용 정량 단서" / "IR 질문지용 텐션 포인트" 두 섹션을 말미에 붙여 다운스트림 스킬이 읽을 수 있게 한다.

## Step 5. Self-review

응답 전 확인:
- [ ] 기사(또는 대체 소스)를 실제로 통독했는가? 접근 실패분은 사유와 함께 명시했는가?
- [ ] 대체 소스 사용 시 최상단 고지를 넣었는가?
- [ ] 기사 속 숫자를 빠뜨리지 않았는가? 단위·기준시점이 붙어 있는가?
- [ ] "영향 있음" 수준에서 멈추지 않고 P/Q/마진/시점까지 내려갔는가?
- [ ] 사실·해석·관측이 섞이지 않았는가? 단독 보도에 신뢰도 태그를 달았는가?
- [ ] 기사 문장을 통째로 복사하지 않았는가?

## 상시 규칙

- 이 스킬은 기사 '해석'까지만. 풀 재무모델은 investment-note/financial-model-builder, 기간 단위 뉴스 대량 수집은 news-collector, IR 질문지 완성본은 ir-question-prep의 몫 — 필요하면 해당 스킬 실행을 제안만 한다.
